### 3.2 Robustez & tratamento de erros

#### 🔴 [Crítico] `save_contestacao` nunca persiste o conteúdo do arquivo
[Backend/App/database.py:529-587](Backend/App/database.py#L529-L587), em particular linhas [Backend/App/database.py:572-573](Backend/App/database.py#L572-L573):

```python
arquivo_nome,         # <- vai para a coluna `arquivo_base` (que era pra ter o conteúdo!)
arquivo_nome,         # <- vai para a coluna `arquivo_base_nome`

A coluna `arquivo_base` foi declarada para receber **o conteúdo base64** (o schema [database.py:276](Backend/App/database.py#L276) tem `arquivo_base TEXT`), mas no INSERT está recebendo o **nome** (mesma string duas vezes). Resultado: o arquivo enviado pelo usuário **nunca chega ao banco** — só o nome dele, duplicado. **Fix:** trocar a primeira linha por `payload.get("arquivo_base_conteudo_base64", "")` (campo já validado em [processo.py:79-103](Backend/App/models/processo.py#L79-L103)).

#### 🟡 [Médio] `usuario_id` aceita `None` virando string vazia
[Backend/App/database.py:565](Backend/App/database.py#L565) — `str(payload.get("usuario_id") or "")` insere `""` na coluna `contestacoes.usuario_id` se a auth falhar silenciosamente. Como a rota usa `Depends(get_authenticated_user)`, isso não deveria acontecer em produção, mas vale uma constraint `NOT NULL` ou `CHECK (usuario_id <> '')` no DDL.

#### 🟡 [Médio] `/health/db` não usa o pool
[Backend/App/database.py:222-229](Backend/App/database.py#L222-L229) e [Backend/App/database.py:143-166](Backend/App/database.py#L143-L166) — `ping_database` chama `_connect()` direto, não `_get_connection()`. Cada healthcheck do orquestrador (a cada 30s no compose) abre/fecha uma conexão TCP+TLS, ignorando o pool. Em escala, isso pode esgotar `max_connections` do Supabase pooler. **Fix:** usar `_get_connection()` para reutilizar.

#### 🟡 [Médio] Sem retry/backoff no `n8n_service`
[Backend/App/services/n8n_service.py:45-59](Backend/App/services/n8n_service.py#L45-L59) — uma falha transitória do n8n (5xx, timeout) cai direto em `N8NServiceError` e o usuário vê 502 imediatamente. Adicionar 1-2 retries com backoff exponencial em status 502/503/504 e timeout reduziria erros falsos.

#### 🟢 Boas práticas
- Pool com lock duplo + lazy init ([database.py:170-201](Backend/App/database.py#L170-L201))
- Cleanup probabilístico de sessões expiradas (2% dos logins) — evita lock prolongado em tabela grande ([database.py:421-442](Backend/App/database.py#L421-L442))

## 5. Revisão do workflow n8n & especificação do novo agente de IA

### 5.1 Caso de uso pedido (edição cirúrgica de .docx base)

A partir do screenshot fornecido, o objetivo do agente quando o usuário envia o formulário é:

**Entrada**
1. Documento `.docx` base — uma contestação já redigida que serve de modelo (ex.: `Contestacao VR Janaina Pereira da Silva.docx`).
2. Instrução textual livre, com chaves explícitas:
   > "modelo base, o que você vai editar nele (somente edite os caracteres que eu mandar):
   > Nome novo: **Erica Cavalcante de Oliveira**
   > Número do processo novo: **0000057-64.2026.5.06.0341**
   > Valor da causa novo se tiver: **27.598,41**
   > Não utilize o LibreOffice"

**Saída esperada (espelhando o ChatGPT do print)**
- Documento editado para download (mesma estrutura, mesma formatação).
- Relatório textual:
  - "Nome antigo `Janaina Pereira da Silva Matos` substituído por **Erica Cavalcante de Oliveira**"
  - "Número do processo `0000091-39.2026.5.06.0341` ajustado para **0000057-64.2026.5.06.0341**"
  - Para campos ausentes, declarar explicitamente: "Não havia valor da causa no documento; caso existisse seria atualizado para **27.598,41**"

**Restrições**
- **Não** usar LibreOffice (sem `soffice --convert-to`).
- Editar **somente** os caracteres especificados (preservar todo o resto: cabeçalho, rodapé, estilos, listas, tabelas, hyperlinks, marcas-d'água).

### 5.2 Gap entre workflow atual e o pedido

### 5.3 Arquitetura proposta

**Não usar LibreOffice headless** (restrição). Não há node n8n nativo que edite `.docx` preservando formatação. Caminho recomendado:

```
Frontend
  ↓ POST /api/editar-contestacao  (FastAPI)
  | body: { arquivo_base64, nome, novo_nome, novo_processo, novo_valor_causa }
  ↓
Backend FastAPI (novo endpoint)
  ├─ valida payload (reusa Processo.validar_arquivo_base64)
  ├─ POST n8n /webhook/editar-contestacao   (novo workflow)
  │   └─ node Code: extrai texto do .docx (python-docx via subprocess
  │      OU passa o base64 + texto extraído pelo backend para o agente)
  │   └─ node Code: chama Claude com instrução para listar antigo↔novo
  │   └─ Webhook responde com pares de substituição
  ├─ recebe pares, aplica substituição com python-docx (no backend)
  ├─ valida ocorrências encontradas vs esperadas
  └─ retorna DOCX editado em base64 + relatório textual
```

**Por que python-docx no backend e não no n8n?** O `.docx` é ZIP+XML. Substituir string em `word/document.xml` quebra quando o texto está fragmentado em múltiplos `<w:r><w:t>` (negrito, sublinhado, hyperlinks). `python-docx` resolve iterando parágrafo a parágrafo e lidando com runs adjacentes. Em n8n via JS é possível com a lib `docxtemplater` ou `mammoth+docx`, mas (1) `n8nio/n8n` não traz essas libs por padrão e (2) o backend já é Python, então o esforço de adicionar dependência é menor lá.

**Dependência nova no backend:** `python-docx==1.1.2` (sem deps extras pesadas).

### 5.4 Prompt do Claude para identificar substituições

Sugestão (system + user para um node Code novo no novo workflow):

```

### 5.5 Validação pós-edição & relatório ao usuário

No backend, após receber `substituicoes` do Claude:

1. Para cada substituição, contar ocorrências reais via `document.xml`. Se `ocorrencias_reais != ocorrencias_esperadas`, **abortar** com erro 422 e mostrar diff ao usuário (impede substituição parcial silenciosa).
2. Aplicar substituição com `python-docx`:
   ```python
   for paragraph in doc.paragraphs:
       for run in paragraph.runs:
           if antigo in run.text:
               run.text = run.text.replace(antigo, novo)
   ```
   (versão simplificada — em produção, agrupar runs adjacentes para casos onde o "antigo" cruza fronteiras de run).
3. Salvar o doc num `BytesIO`, base64-encode, retornar ao frontend.
4. Anexar à resposta o relatório no formato do print:
   ```json
   {
     "status": "ok",
     "arquivo_editado_base64": "...",
     "arquivo_editado_nome": "Contestacao_editada_<numero_novo>.docx",
     "relatorio": [
       "Nome antigo \"Janaina Pereira da Silva Matos\" substituído por **Erica Cavalcante de Oliveira**.",
       "Número do processo \"0000091-39.2026.5.06.0341\" ajustado para **0000057-64.2026.5.06.0341**.",
       "Não havia valor da causa especificado no documento; caso existisse teria sido atualizado para **27.598,41**."
     ]
   }
   ```

### 5.6 Revisão do workflow atual `contestacao-claude` (em segundo plano)

Auditoria do JSON em [docs/n8n_workflow_contestacao_claude.json](docs/n8n_workflow_contestacao_claude.json):

#### 🟢 Pontos bons
- **Validação CNJ no node Code** (`/^\d{7}-\d{2}\.\d{4}\.\d\.\d{2}\.\d{4}$/`) replica a regex do backend ([processo.py:10](Backend/App/models/processo.py#L10)) — defesa em profundidade.
- **Sanitização de input para Supabase REST**: `SAFE_TIPO_RX` e `SAFE_NUMERO_RX` rejeitam caracteres fora do esperado antes de concatenar na URL — bom contra PostgREST injection.
- **Timeout de 8s** na consulta Supabase, com `AbortController`.
- **Prompt-caching `ephemeral`** ativo: `'anthropic-beta': 'prompt-caching-2024-07-31'` + `cache_control: { type: 'ephemeral' }` no `system`. Como o `SYSTEM_PROMPT` é estável, cada chamada após a primeira (dentro de 5min) economiza tokens.
- **Fallback determinístico** completo: se a API Claude falhar (sem chave, timeout, parse error, HTTP non-2xx), entrega minuta básica em vez de erro 500.
- **Limites de tamanho** para cada campo do JSON de resposta (`clipStr`) — protege contra retorno gigante do modelo.

#### 🟡 Pontos de atenção
- **`MAX_TOKENS=6000` agora, antes era 12000** (ver descrição: "Claude (12k tokens)"). Mismatch entre descrição e código atual. Atualizar descrição.
- **`MIN(TIMEOUT_MS, 45000)` no fallback** ([n8n_workflow_contestacao_claude.json](docs/n8n_workflow_contestacao_claude.json)): `Math.max(10000, parseInt(...) || 45000)` — se o env não estiver setado o default vira 45s, mesmo que o env do compose passe `120000`. Não é bug grave porque o Math.max protege, mas tornar consistente com o env (`120000` em [docker-compose.yml:92](docker-compose.yml#L92)).
- **N8NResponse no backend não cobre `engine_ia` nem `auditoria` em testes** — só está no model com `Any`. Faria sentido tipá-los para evitar surpresas (relacionado às falhas em 2.1).
